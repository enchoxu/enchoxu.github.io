Prometheus进程消耗物理内存

prometheus是基于go语言开发的监控指标收集处理系统，在discovery服务达1k万instances时候，实际物理内存消耗达到40+G，这部分资源具体用到哪里了呢，我们进一步分析：

从PProf分析内存

为了能够观察这47G内存的使用分布，我们开启pprof监控进一步观察

pprof/heap表示可以查看堆内存的使用情况，Type类型有：

inuse_space(正在使用的objects字节数)、

inuse_objects（在使用的objects个数）、

alloc_objects（自进程启动之后已分配objects个数）、

alloc_space（自进程启动之后已分配objects的字节数）

备注：alloc_objects和inuse_objects有什么区别呢，alloc_objects指自进程启动起分配过的object个数，有可能该object或者在使用（即inuse_objects), 有可能待回收的object。inuse_objects则表示当前正在使用的objects个数

默认是查看inuse_space, 用top、list [Func]可以查看具体分配

可以调用list uvarintStr可以查看具体分配内存的源码：

暂且不看这里使用是否合理，我们先看看top使用总内存29.11GB，那么剩下的47-29.11=16G内存消耗在哪里呢？为了继续寻找答案，我们在打印更多的MemStat信息：

这个就是运行时（runtime)的内存信息，我们先一一解释下各个字段的含义：(感兴趣可以看https://github.com/golang/go/blob/master/src/runtime/mstats.go源码MemStats struct结构说明)

Alloc：已分配的堆objects字节数（和HeapAlloc含义一样）

TotalAlloc：已分配的堆objects累计字节数，该值持续增加

Sys：从OS获得的内存的总字节数，等于HeapSys+GCSys+OtherSys+xxxSys的总和，衡量存放堆、栈和其他内部数据结构使用的内存空间，在某个时刻这些虚拟地址空间不会全部真正被映射到物理地址。

Mallocs：指累计已分配的堆里objects个数，活跃的objects个数=Mallocs-Frees

Frees: 指累计已释放的堆里objects个数

HeapAlloc: 指已分配objects的字节数。这里也包括还未被内存回收回收掉，且未使用的object。

HeapObjects：指已分配objects的个数

HeapSys：指从OS申请给堆内存使用的字节数。

HeapIdle： 指空闲的span字节数（首次提到span，暂且理解为一个span有 pages组成，里面存放objects），这部分内存可以重复被运行时使用，或者回收。

HeapInuse：指至少有一个object在使用span空间的字节数，以span为单元计算，类似于集装箱。

HeapReleased：表示在HeapIdle空闲的span里，哪些已经是物理内存回收掉的字节数。

Stack： StackInuse(在使用的) /StackSys( 已申请的StackSys)

MSpan: MSpanInuse / MSpanSys

MCache: MCacheInuse / MCacheSys

BuckHashSys: is bytes of memory in profiling bucket hash tables.

GCSys:存放垃圾回收的元数据

OtherSys：非堆申请的其他内存空间

NextGC：下次进行gc回收的目标堆大小

LastGC： 上次进行gc的时间戳

因此，从OS申请的内存= 堆申请的内存+栈申请的内存+Mspan管理的内存+MCache管理的内存+hash表申请的内存+gc回收使用内存+其他申请的内存空间

这里数据可以肯定，申请堆内存占了大头（46G/49G约占了93%)，这批内存的使用分配情况如下：

HeapSys(46G) = HeapIdle (8G) + HeapInuse(38G)

即在抓取的某个时刻，有8G的空闲span+36G至少有一个object的span空间。 这8G内存由go运行时垃圾回收定时优化，38G已使用span空间里面，有34G已分配的objects（即HeapAlloc)，装箱率达到(34/36)94%。

这34G申请的objects空间, 在使用的约29.11G,另外有4.89G的object未被回收.

结论：因此可以得出，真正在使用的除了29.11G，其他内存占用情况如下：

4.89G objects已使用未回收+ 4G span内存有额外空闲objects + 8G 空闲span =14.89G

这里的objects已使用未回收和span内有额外空闲objects怎么理解呢？我们继续往下看go运行时内存是如何管理的。

go运行时内存管理

基本概念

golang摒弃了传统的内存管理，改成自行管理，每次分配时不需要每次系统调用申请内存。

内存分配器在管理内存块分两种关键对象：

span：由多个地址连续的页（pages）组成的大内存块。

object: 将span按照特定大小切分成多个块(object)，每个块可以存储一个对象

分配器由三种组件组成（主要是管理和分配span和object）：

Cache: 每个运行期工作线程会分配一个cache，用于每个P线程无锁分配object；

Central：为Cache提供切分好的span资源；

Heap：管理限制span，需要时向操作系统申请新内存；

分配流程：

  1. 计算待分配对象的规格(size class)
  2. 从cache.alloc 数组找到规则相同的span
  3. 从span.freelist链表取可用的object
  4. 如果span.freelist为空，从central获取新的span
  5. 如果central.nonempty为空，从heap.free/freelarge获取，并切成obect链表
  6. 如果heap没有大小合适的闲置span，向系统申请内存

Openssl 知识

一、为什么需要加密：

防止网络窃听行为；

加密技术（cryptography)

机密性：confidentlity

二、对称加密（symmetric cryptography)

在加密和解密时使用相同的密钥。目前广泛使用AES。

对称密码的多种别名：

公共密钥密码：common-key cryptography

传统密码：conventional cryptography

私钥密码： secret-key cryptography

共享密钥密码：shared-key cryptography

三、非对称加密

目前广泛使用的：RSA，Elgamal，Rabin，DH，ECDH

四、单行散列函数

为了防止被篡改，有意识的给发布的软件发布一个散列值hash。散列值是用单向散列函数计算出来的值。目前广泛使用的SHA-2(SHA-224,SHA-356,SHA-384,SHA-512)和SHA-3（Keccak算法）

散列值，又被称为Hash值、密码校验和（cryptography checksum）、指纹：fingerpritnt、消息摘要（message digest)

散列值保证了数据的完整性（intergrity)

五、消息认证码

为了确认消息来自期望的通讯对象，引入消息认证码（message authentication code)

消息验证码最常用的单向散列函数是HMAC

六、数字签名

数字签名类似现实生活中的签名和盖章，防止用户抵赖，伪装，伪造，篡改的密码技术。

签名算法包括：RSA、Elgamal、DSA、椭圆曲线DSA（ECDSA）、爱德华兹曲线DSA（EDDSA）

参考：https://halfrost.com/cryptography_overview/

关于Https通信详解

1.密码学的基础知识

加密算法大致分两类：

- 不基于key的加密算法。
  • 基于key的加密算法：
    • 对称加密： 比如DES,AES，通信双方一方用key加密之后，另一方用相同的key解密
    • 非对称加密：著名的RSA算法，加密的时候有一个公钥和一个私钥，公钥可以交给对方，a给b发送信息，a用b的弓腰加密，b用自己的私钥解密。反之，b给a发送信息，b用a的公钥加密，a用自己的私钥解密；

2 网络安全问题

日常生活中，我们上网用到最多的应用层协议是http协议，不过http协议不使用TLS/SSL进行加密通信。所有信息都是明文传播的，带来了三个大的风险：

  1. 窃听：第三方可以获知通信内容；
  2. 篡改：第三方可以修改通信内容；
  3. 冒充：第三方可以冒充他人身份参与通信。

2.1 如何防止窃听

为了保证数据的机密性，通常的方式是对数据通过某种算法进行加密，大家协商好加密协议后，用同等算法进行解密；通用都是基于key对加密算法。但问题是如何将key传送给对方而不被第三方窃听到key呢？

2.2 如何防止篡改

为了防止加密数据被第三方窃取到并篡改，通常采取的做法是增加特征码，机发送方利用对称密钥方式对手中的特征码进行加密，接收方利用相同的密钥对手中的特征码进喜解密，从而确认特征码是否一致。

2.3 如何防止身份冒充

以上信息只是解决数据的交换获取问题，但是网络的身份验证问题依然没有解决，换句话说，通讯双方的确可以获取统一的加密信息了，并且是通过相对安全的方式获取得知的，但是通讯双方在交换获取密钥前，又怎么确认得知，双方的确是要进行通信的发送方和接收方呢？

利用非对成加密算法和数字证书的可靠性。

非对称加密算法遵循一个基本原则：公钥加密的只能利用与之配对的私钥进行解密。

3 数字证书

3.1 数字证书的由来

通过非对称加密加密双方协商的对称加密的key，可以保证对称加密的安全性，但如果中间有人冒充接受方伪造公钥和发送方进行协商通信，会导致数据的窃取。所以我们需要一种机制可以识别公钥的发送方的真实性。这就是证书的由来：

  1. A和B端首先生成自己的公约和私钥的密钥对，为了使对方能相信自己的公约信息，将自己的弓腰信息告知第三方证书机构，利用第三方机构对自己的公钥进行公证。第三方机构会制作一个数字证书(.crt)，并且第三方机构也要给自己设置一个合法的公钥和私钥，并且公钥设为第三方机构的公钥证书。
  2. A和B获得公正过的证书信息，并通过证书信息传递，得到对方的公钥。
  3. A和B通过第三方公钥解密数字签名，证明证书的真实身份。

3.2数字证书的签发

3.2.1 数字证书的组成

目前SSL、TLS的证书管理机制均使用x509的格式。数字证书（如server.crt)由两部分组成：

C： 证书相关信息

​ 服务器的公钥以及证书的过期时间

​ 证书的合法拥有人信息

​ 证书的使用说明

​ 证书颁发机构（CA)

​ 证书的签名算法

​ 证书签名的校验码

S: 证书的数字签名(由CA证书通过加密算法生成的)

3.2.2 根CA证书(ca.crt)

一般，数字证书是从授信的权威证书机构（CA）买来的，浏览器里面一般会内置一些权威的CA，在使用https的时候，只要这些CA签发的证书，浏览器都是可以认证的，要是在与服务器通信的时候，收到一个没有权威CA认证的证书，就会提示不受信任证书的错误。

在上面的项目中，通过openssl自己签发一个ca根证书，之后使用这个根证书签发一个服务器的数字证书server.crt和服务器的私钥server.key给服务器，在客户端和服务器通信的时候，要指定ca根证书，作用就相当于浏览器内置的哪些行为权威证书一样，用于进行服务器的身份校验。

4.HTTPS通信过程详解

Https通信过程结合了对称加密和非对称加密的两种方法。Https服务端在链接建立过程（ssl 握手协议中），会将自身的公钥发送到客户端。客户端拿到公钥后，通过对称加密与服务器协商数据的对称加密密钥。

4.1客户端发出ClientHello请求

在这一步，客户端主要想服务器提供以下信息：

1) 支持的协议版本，比如TLS1.0版本。

2）一个客户端生成的随机数a，稍后用于生成”对话密钥”

3）支持的加密方法，比如RSA公钥加密

4）支持的压缩方法

4.2 服务器回应ServerHello

回应的内容包括：

1）确认使用的加密通信协议版本，比如TLS1.0

2）一个服务器生成的随机数b，稍后用于生成“对话密钥”

3）确认使用的加密算法

4）数字证书

4.3 客户端回应

首先验证服务器证书，如果证书不是可信颁发机构、或者证书域名不一致，或者证书过期，会提示一个经过。

如果证书ok，客户端会从证书取出服务器的公钥，并向服务器发送以下三项信息：

1）公钥

2）协商好的加密算法

3）握手通知

4.4 服务器最后的回应

1）服务器握手结束通知
